## 3. Important Execution Policy

### 3.1 Do Not Use Dummy Declarations

Never silence PJASS with fake declarations such as:

```jass
integer HASH_ABILITY = 0
integer yd_MapMaxX = 0
function MissingThing takes nothing returns nothing
endfunction
```

Only emit a symbol if one of these is true:

```text
- It exists in source and was missed by parser/lowering.
- It is generated by the compiler as part of a known lowering rule.
- It is explicitly provided by a documented environment file or validation environment.
- It exists in the JassHelper output and its provenance is understood.
```

### 3.2 Do Not Add Default Returns to User Source Functions Blindly

PJASS reports `returnMissingValue` for functions with non-void returns that do not return on every path.

Do **not** automatically patch user-authored source functions like:

```jass
function Foo takes nothing returns integer
    if condition then
        return 1
    endif
endfunction
```

with:

```jass
return 0
```

unless legacy JassHelper behavior confirms this is required and safe.

Allowed:

```text
- Add missing returns to compiler-generated wrappers.
- Add missing returns to adapter functions generated by vjassc.
- Fix lowering bugs that accidentally removed/failed to emit a return.
- Preserve source missing-return issues as source-policy diagnostics if legacy behavior is uncertain.
```

### 3.3 Raw `code` Callback Adapters Must Be Context-Aware

JASS raw `code` callback positions cannot safely receive arbitrary parameterized lambdas.

Do not generate adapters that invent parameter values.

Allowed only when the parameter source is known:

```text
- function-interface dispatch argument globals
- explicit compiler-generated callback wrapper globals
- known event payload extraction
- known previously-bound callback context
```

Unknown parameterized lambda to raw `code` should remain a classified blocker or diagnostic, not a silent unsafe adapter.

### 3.4 Performance Optimizations Must Be Low-Risk

Allowed now:

```text
- memoization
- precomputed maps
- replacing hot regex with scanners
- skipping scans on impossible lines
- combining multiple full-output passes into one
- reusing existing output for validation
- separating expensive validation from normal codegen
```

Not allowed yet unless PJASS passes:

```text
- large IR redesign
- full codegen rewrite
- removing diagnostics
- removing validation paths
- changing symbol semantics solely for speed
```

## 4. Anti-Inefficiency Workflow

The previous workflow became inefficient because Codex was using low thresholds such as `grouped <= 500` and repeatedly running full PJASS after tiny changes.

Phase 12 must use this bounded workflow.

### 4.1 Batch First, Full PJASS Second

For each blocker group:

```text
1. Analyze existing PJASS stdout using --analyze-pjass-log.
2. Extract 5-20 representative examples using --emit-pjass-examples.
3. Write focused fixtures.
4. Implement a batch fix for the whole class.
5. Run fixture tests.
6. Run syntax-lite on existing or generated output.
7. Run full PJASS only at batch checkpoint.
```

Do not run full PJASS after every minor line edit.

### 4.2 Full PJASS Run Budget

Recommended maximum full real-sample PJASS runs for this phase:

```text
Run 1: baseline confirmation after pulling latest master
Run 2: after returnMissingValue / statement-shape batch
Run 3: after raw code callback adapter batch
Run 4: after undefined/provenance / forward-cycle batch
Run 5: final acceptance run
```

If a run reduces grouped count by fewer than 5 and does not unblock a specific blocker, stop iterating that class and move to the next one.

### 4.3 Use Offline Tools Aggressively

Use these before full reruns:

```bash
build/vjassc --analyze-pjass-log build/input.phase11.pjass.validation.pjass.stdout.txt \
  --emit-validation-report build/input.phase12.triage.json \
  --emit-pjass-examples 50

build/vjassc --validate-existing-output build/input.phase11.pjass.out.j \
  --emit-validation-report build/input.phase12.existing.validation.json \
  --check-output-syntax-lite
```

If these tools are insufficient, extend them before brute-force full runs.

### 4.4 Stop Conditions

Stop a blocker batch if:

```text
- The batch needs dummy declarations.
- The batch needs unsafe default returns in source functions.
- The batch requires guessing callback arguments.
- The batch creates syntax-lite regressions.
- The batch increases grouped PJASS count by more than 10.
```

## 8. Task C: Raw `code` Callback Adapter

Current baseline:

```text
callbackCodeSignatureMismatch: 30
```

### C.1 Understand the Problem

PJASS errors occur when a lambda/function with parameters is passed to a raw `code` position.

Unsafe:

```jass
call SomeCallback(function vjlambda__108)
```

where:

```jass
function vjlambda__108 takes integer x returns nothing
```

Raw callback positions often require:

```jass
function Callback takes nothing returns nothing
```

or for conditions:

```jass
function Callback takes nothing returns boolean
```

### C.2 Classify Contexts

Classify call sites into:

```text
1. TimerStart-like action callback: takes nothing returns nothing
2. TriggerAddAction-like callback: takes nothing returns nothing
3. Condition/Filter-like callback: takes nothing returns boolean
4. function-interface dispatch callback: args are in compiler globals
5. custom API callback with known signature
6. unknown raw code position
```

### C.3 Generate Adapter Only When Safe

Safe adapter example:

```jass
function vjlambda__108 takes integer x returns nothing
    ...
endfunction

function vjlambda__108_code_adapter takes nothing returns nothing
    call vjlambda__108(vjassc__knownArg1)
endfunction
```

But only if `vjassc__knownArg1` is genuinely known.

For no-arg lambda accidentally classified as mismatch, just use the lambda directly.

For boolean condition adapters:

```jass
function vjlambda__108_condition_adapter takes nothing returns boolean
    return vjlambda__108(vjassc__knownArg1)
endfunction
```

### C.4 Unknown Context Policy

If argument provenance is unknown:

```text
- Do not generate adapter.
- Keep classified PJASS issue.
- Add validation report entry:
  callbackCodeSignatureMismatch.unknownContext
```

### C.5 Tests

Add fixtures:


## 11. Task F: Forward Function Cycles

Current baseline:

```text
forwardFunctionReference: 3
```

Known cycles:

```text
- SyncBus onInit <-> onDataSync
- CombineSession buildSelector <-> runStage
- MopUpItem lambda <-> MopUpItemCreate
```

### F.1 Classify Signatures

For each cycle, capture:

```text
caller function
callee function
parameter types
return type
is generated or source
is no-arg/no-return
is called in expression position
is called as statement only
```

### F.2 Safe Strategy Matrix

Use this strategy table:

```text
Case A: takes nothing returns nothing, statement position
    -> can use ExecuteFunc("callee") bridge if runtime behavior is acceptable.

Case B: has parameters or returns value
    -> cannot use simple ExecuteFunc.
    -> consider function-interface-like dispatcher globals only if safe and source-equivalent.

Case C: generated lambda cycle
    -> prefer reordering/renaming generated helper if possible.
    -> if impossible, use generated dispatcher.

Case D: user source mutual recursion with return values
    -> preserve as classified source limitation unless legacy JassHelper has a clear bridge model.
```

### F.3 Do Not Break Runtime Behavior

Do not replace return-value calls with `ExecuteFunc`.

Do not discard arguments.

Do not use global temp dispatch unless reentrant behavior is acceptable or matches JassHelper style.

### G.4 Optimize `emitFunctions`

Likely causes:

```text
- repeated full-line lowering
- repeated symbol lookups
- repeated struct/member scans
- repeated lambda extraction
- repeated regex over lines with no relevant syntax
```

Implement:

```text
- Precompute per-line feature flags:
    hasDot
    hasBracket
    hasFunctionKeyword
    hasLambdaSyntax
    hasCall
    hasSet
    hasStructCandidate
- Skip expensive lowering passes when feature flags rule them out.
- Cache function signature lookup by name.
- Cache struct lookup by source and generated names.
- Avoid running receiver-chain scanner on lines without "." or "[".
- Avoid repeated lowerExpression on identical generated helper lines where context-independent.
```

### G.5 Optimize `emitStructSupport`

Implement:

```text
- Generate each struct support block exactly once.
- Precompute struct support function names.
- Precompute field/method generated names.
- Avoid scanning all structs for each support method.
- Avoid re-lowering support function bodies if already generated.
```

### G.6 Optimize `sanitizeOutput` and Validation Prep

Implement:

```text
- Combine multiple output sanitation scans into one scanner pass where safe.
- Avoid full-output regex passes.
- Skip syntax-lite buckets unless requested.
- For --validate-existing-output, do not rebuild program or rerun codegen.
- For --analyze-pjass-log, never load/parse input source unless explicitly requested.
```

### G.7 Optimize `lowerLambdas`

Implement:

```text
- Extract lambdas once per function body.
- Store processed body in cache.
- Avoid rescanning generated lambda functions for lambdas.
- Precompute capture candidates per function.
```

### G.8 Performance Regression Guard

Add a perf regression warning:

```text
If full validation total > previous phase total + 10%, print warning.
If codegen > previous phase codegen + 10%, print warning.
```

Do not fail tests on perf warning yet.

### Acceptance

```text

- syntax-lite still green
- grouped PJASS count does not increase
- no fixture regression
```
